1. 项目背景与问题

APS 系统的 order_repay_withhold 表为 MySQL 5.6 表，当前面临严重的存储和性能问题：

指标当前值影响
存储空间超过 800G磁盘容量告警，影响整体数据库性能
数据行数2.8 亿查询性能下降，定时任务扫描耗时长
MySQL 版本5.6不支持分区表优化等新特性

改造目标

  1. 数据冷热分离：最近两个月的数据为热数据，存储在 MySQL 新表中；冷数据（全量）存储在 OceanBase 中
  2. 写入层面：前期三写（旧MySQL + 新MySQL + OB），迁移完成后改为双写（新MySQL + OB）
  3. 读取层面：迁移前查旧表，迁移后先查新热表，miss 则查 OB
  4. 数据迁移：将三写开启前的历史数据迁移到 OB，迁移过程中保持数据一致
关键决策记录 冷热边界按 date_created 划界 · 新热表与旧表同库同实例（共享DataSource，天然支持本地事务强一致）· OB首次接入APS · OB写入为事务提交后异步MQ · 迁移任务为APS内置定时任务 · 写入模式枚举开关（OLD_ONLY/TRIPLE/DUAL）+ 读取开关（readNew），共2个开关 · 路由逻辑内聚在现有ServiceImpl中，上层调用方零改动

3.2 三阶段划分

1 三写期

写入：旧MySQL + 新MySQL（本地事务）+ OB（先同步写，失败走MQ）

读取：只查旧MySQL

触发：writeMode=TRIPLE, readNew=false

2 迁移期

写入：同三写期（旧MySQL + 新MySQL + OB）

读取：只查旧MySQL

触发：writeMode=TRIPLE, readNew=false

3 切读期

写入：同三写期（旧MySQL仍在写）

读取：先查新MySQL，miss则查OB

触发：writeMode=TRIPLE, readNew=true

4 双写期

写入：新MySQL + OB（先同步写，失败走MQ）

读取：先查新MySQL，miss则查OB

触发：writeMode=DUAL, readNew=true

注意
  • 阶段一和阶段二在写入层面相同（三写），区别在于阶段二同时运行迁移任务
  • 阶段三（切读期）是关键过渡阶段：写入仍走三写（旧表仍在写），但读取切到新热表+OB。如果切读后发现问题，可以秒级回滚到读旧表（readNew=false），旧表数据仍在、写入仍在，回滚零风险
  • 切读期观察一周左右确认无问题后，再停写旧表进入双写期

3.3 Apollo开关设计

用一个写入模式开关 + 一个读取开关 + 一个OB写入日志开关 + 一个OB读取日志开关，共4个开关覆盖所有场景：

开关类型默认值说明
aps.apollo.config.repayWithholdWriteMode String OLD_ONLY 写入模式：OLD_ONLY / TRIPLE / DUAL
aps.apollo.function.switch.repayWithholdReadNew boolean false 读取路由开关，true=先查热表miss查OB
aps.apollo.function.switch.repayWithholdObWriteLog boolean false OB写入日志开关，true=打印OB同步写入/降级MQ的详细日志
aps.apollo.function.switch.repayWithholdObReadLog boolean false OB读取日志开关，true=打印查OB的OUTREQ/OUTRESP日志，用于观察穿透到OB层的查询量

写入模式枚举定义

public enum RepayWithholdWriteModeEnum {
    OLD_ONLY("OLD_ONLY", "只写旧MySQL表"),
    TRIPLE("TRIPLE", "三写：旧MySQL + 新MySQL热表 + OB异步"),
    DUAL("DUAL", "双写：新MySQL热表 + OB异步（停写旧表）");
}

各模式对应的写入行为

writeMode旧MySQL新MySQL热表OB（异步）readNew说明
OLD_ONLY ✅ ❌ ❌ false 默认状态，代码上线后无任何行为变化
TRIPLE ✅ ✅ ✅ false 三写：旧+新（本地事务强一致）+ OB异步
DUAL ❌ ✅ ✅ true 双写：停写旧表，读写走新表+OB
紧急回滚 将 writeMode 从 TRIPLE/DUAL 切回 OLD_ONLY，readNew 切回 false，即可秒级回滚到只写旧表的状态。

4.1 在现有ServiceImpl中内聚路由逻辑

不新增路由层Service，直接在现有 OrderRepayWithholdServiceImpl 中增加路由逻辑。新增注入 HotDao 和 OB 异步写入组件，每个方法内部通过 Apollo 开关判断走哪条路径。所有上层调用方零改动，仍然注入 orderRepayWithholdServiceImpl。

上层调用方（无需改动，仍然注入 orderRepayWithholdServiceImpl）
  │

// saveRepayInfo 改造
@Override
@Transactional(rollbackFor = Exception.class)
public int saveRepayInfo(OrderRepayWithholdEntity entity,
                         OrderRepayWithholdDetailEntity detailEntity) {
    RepayWithholdWriteModeEnum writeMode = getWriteMode();
    int countRepay = 0;
    int countDetail = 0;

    // 写旧表（OLD_ONLY / TRIPLE），insertSelective后MyBatis自动回填自增ID到entity.id
    if (writeMode == OLD_ONLY || writeMode == TRIPLE) {
        countRepay = orderRepayWithholdDao.insertSelective(entity);
    }

    // 写新热表 + 写OB（TRIPLE / DUAL）
    if (writeMode == TRIPLE || writeMode == DUAL) {
        // TRIPLE：entity.id已被旧表回填，热表insertSelective使用该id，三表id一致
        // DUAL：旧表不写，由热表自增生成id并回填到entity.id
        orderRepayWithholdHotDao.insertSelective(entity);
        // OB写入：事务提交后先同步写OB，异常时降级为异步MQ
        OrderRepayWithholdEntity entityForOb = cloneEntity(entity);
        registerAfterCommitCallback(() ->
            writeObWithFallback(() ->
                orderRepayWithholdObDao.insertOrupdateByRepayTranNo(entityForOb),
                () -> obWriteProducer.sendInsertMessage(entityForOb),
                "INSERT", entityForOb.getRepayTranNo()));
    }

    // detail表与withhold表同库同数据源，所有模式下都写入
    countDetail = orderRepayWithholdDetailDao.insertSelective(detailEntity);

    return countRepay > 0 && countDetail > 0 ? 1 : 0;
}

自增ID一致性保证
  • MyBatis 的 insertSelective 配合 useGeneratedKeys=true，插入后自动将数据库自增ID回填到 Entity 的 id 字段
  • TRIPLE 模式：旧表先插入 → id 回填到 entity → 热表插入时 entity 已有 id（热表 Mapper 的 <if test="id != null"> 使用传入 id）→ OB 异步写入时 entity 中已携带 id，三表 id 一致
  • DUAL 模式：旧表不写入 → 热表插入 → id 回填到 entity → OB 异步写入时使用热表回填的 id
  • 热表 Mapper XML 中 insertSelective 的 id 字段需使用 <if test="id != null"> 动态判断：id 为 null 时由数据库自增（DUAL 模式），id 不为 null 时使用传入值（TRIPLE 模式下由旧表回填）

4.2 事务一致性保证

强一致：旧MySQL + 新MySQL

新热表与旧表在同一个 MySQL 实例的同一个库中，共享同一个 DataSource。Spring 的 @Transactional 基于 ThreadLocal 绑定 Connection，当方法有 @Transactional 时，旧表 DAO 和新热表 DAO 的操作使用同一个 Connection，天然属于同一个事务。

调用链示例：
OrderRepayWithholdFlowServiceImpl.updateAndSaveRepayWithholdBill()  ← @Transactional 开启事务
  → orderRepayWithholdService.updateRepayEntityByRepayTranNoAndState()
    → orderRepayWithholdDao.update(...)     ← 同一Connection（旧表）
    → orderRepayWithholdHotDao.update(...)  ← 同一Connection（新热表）
                                                               ↑ 全部成功才commit，任一失败则rollback

嵌套事务安全性

改造后，写入方法自身都加了 @Transactional，而部分调用方也有 @Transactional，形成嵌套。Spring 默认 PROPAGATION_REQUIRED 的语义是"有事务就加入，没有就新建"，在同 DataSource 场景下：

场景行为强一致
调用方有事务（如 updateAndSaveRepayWithholdBill） 内层加入外层事务，共享同一个 Connection ✅ 旧表+热表在同一事务中
调用方没有事务（如 AbstractPartnerRepayWithholdProcessService） 自身事务生效，新建独立事务 ✅ 旧表+热表在同一事务中
多层嵌套（如 FlowService → ServiceImpl.saveRepayInfo） 所有内层加入最外层事务，共享同一个 Connection ✅ 全部操作在同一事务中

关键前提：旧表和热表在同一个 MySQL 实例同一个库，共享同一个 DataSource，因此同一个事务中操作两张表使用的是同一个 Connection，不存在跨数据源事务问题。

4.3 OB异步写入机制

事务提交后回调的触发时机

registerAfterCommitCallback 通过 TransactionSynchronizationManager 注册，回调绑定在当前活跃的物理事务上。当存在事务嵌套时（内层 PROPAGATION_REQUIRED 加入外层事务），回调在最外层事务提交后才触发。

场景：OrderRepayWithholdFlowServiceImpl.updateAndSaveRepayWithholdBill()（外层事务）
  → orderRepayWithholdService.updateRepayEntityByRepayTranNoAndState()（内层，加入外层事务）
    → 旧表 update
    → 热表 update
    → registerAfterCommitCallback(OB写入)  ← 注册到外层事务上
  → orderRepayWithholdBillService.insert()（bill表）
  → 外层事务 commit  ← 此时触发 afterCommit，发送OB写入MQ

OB写入在所有MySQL操作（withhold旧表 + withhold热表 + bill表）都成功提交后才执行

这保证了 OB 异步写入的前提是所有 MySQL 写入都已持久化，不会出现 MySQL 写入回滚但 OB 已写入的不一致情况。

private void registerAfterCommitCallback(Runnable callback) {
    if (TransactionSynchronizationManager.isSynchronizationActive()) {
        TransactionSynchronizationManager.registerSynchronization(
            new TransactionSynchronizationAdapter() {
                @Override
                public void afterCommit() {
                    callback.run();
                }
            });
    } else {
        // 无事务上下文时直接执行
        callback.run();
    }
}

MQ异步写入流程

本地事务（旧MySQL + 新MySQL）提交成功
  → TransactionSynchronizationManager.afterCommit()
    → 发送 RocketMQ 消息（topic: aps_ob_write_topic）
      → OB 消费端执行写入
        → 成功：ACK
        → 失败：RocketMQ 自动重试（默认16次）
          → 超过最大重试：进入死信队列，人工处理

关键设计决策

  • 全量写入而非增量：OB 的更新操作使用全量 Entity 覆盖（updateByRepayTranNo，不带乐观锁条件），避免旧状态值在 OB 侧不一致导致更新失败
  • 插入幂等：OB 的 insert 使用 INSERT ON DUPLICATE KEY UPDATE（OB 支持 MySQL 协议），保证重复消费不报错
  • 无额外补偿表：OB 写入失败通过 RocketMQ 重试机制保证最终一致，超过重试次数进入死信队列人工处理

6. 数据迁移任务

6.1 迁移任务设计

新增定时任务 OrderRepayWithholdMigrationTask，负责将三写开启前的历史数据从旧MySQL表迁移到OB表。

迁移范围 只迁移三写开启时间点之前 date_created 的数据。三写开启后的数据已通过三写机制写入OB，无需重复迁移。

迁移策略：基于 ID 范围分批迁移，与现有 OrderRepayWithholdQueryTask 类似。

@Slf4j
@Service("orderRepayWithholdMigrationTask")
public class OrderRepayWithholdMigrationTask
        extends AbstractCronTaskClientService {

    @Resource
    private OrderRepayWithholdDao orderRepayWithholdDao;                  // 旧表
    @Resource
    private OrderRepayWithholdObDao orderRepayWithholdObDao;              // OB表
    @Resource
    private OrderRepayWithholdMigrationRecordService migrationRecordService;

    @Override
    public List<ExecuteItem> selectTasks(TaskParam taskParam, String ownSign) {
        // 1. 获取迁移起始ID（从迁移进度表读取）
        Long startId = migrationRecordService.getLastMigratedId();

        // 2. 获取三写开启时间点前最大的ID（迁移终点）
        Long endId = orderRepayWithholdDao
            .selectMaxIdBeforeTripleWriteTime();

        // 3. 分批读取旧表数据并写入OB
        long batchStartId = startId;
        long batchSize = 10000;  // 每批1万条
        while (batchStartId <= endId) {
            long batchEndId = batchStartId + batchSize;

            List<OrderRepayWithholdEntity> batch =
                orderRepayWithholdDao.queryByIdRange(batchStartId, batchEndId);

            if (CollectionUtils.isNotEmpty(batch)) {
                // 批量写入OB（INSERT ON DUPLICATE KEY UPDATE 保证幂等）
                orderRepayWithholdObDao.batchInsert(batch);

                // 更新迁移进度
                long lastId = batch.get(batch.size() - 1).getId();
                migrationRecordService.updateLastMigratedId(lastId);
            }

            batchStartId = batchEndId;
            ThreadUtil.sleep(100);  // 批次间休眠，降低对旧表压力
        }
        return null;
    }
}

6.2 迁移一致性保证

问题：操作时间线为"先开启三写 → 再启动迁移任务"。迁移任务只迁移三写开启前创建的数据（date_created <= migrationStartTime），但这些数据在三写开启后可能被更新（代扣状态流转），如何保证迁移到 OB 的数据是最新的？

方案：二次校验——全量迁移完成后，按 ID 范围扫描旧表中三写开启前创建的记录，逐条与 OB 中对应记录比对关键字段（withhold_state、withhold_amount、result_code、result_msg、date_updated 等），关键字段不一致时以旧表数据为准覆盖写入 OB。

为什么不按 date_updated 查询 旧表无 date_updated 索引，按 date_updated >= migrationStartTime 查询会全表扫描，2.8 亿行遍历效率极低。改为按 ID 范围扫描（走主键索引），逐条比对关键字段，性能可控。

二次校验实现

全量迁移完成后，按 ID 范围分批扫描三写开启前的记录，逐条与 OB 比对关键字段，不一致时以旧表数据覆盖写入 OB：

// 二次校验：按ID范围扫描，逐条比对关键字段，不一致时以旧表为准覆盖OB
Long maxIdBeforeMigration = orderRepayWithholdDao
    .selectMaxIdBeforeTripleWriteTime(migrationStartTime);
long batchSize = 10000;
long startId = 0;
int diffCount = 0;

while (startId <= maxIdBeforeMigration) {
    long endId = startId + batchSize;

    // 按ID范围查询旧表数据（走主键索引，性能可控）
    List<OrderRepayWithholdEntity> batch =
        orderRepayWithholdDao.queryByIdRangeAndCreatedBefore(
            startId, endId, migrationStartTime);

    if (CollectionUtils.isNotEmpty(batch)) {
        for (OrderRepayWithholdEntity mysqlEntity : batch) {
            // 查OB对应记录
            OrderRepayWithholdEntity obEntity =
                orderRepayWithholdObDao.selectByRepayTranNo(
                    mysqlEntity.getRepayTranNo());

            if (obEntity == null) {
                // OB缺失，以旧表为准写入
                orderRepayWithholdObDao.insertOrupdateByRepayTranNo(mysqlEntity);
                diffCount++;
                continue;
            }

            // 比对关键字段，不一致时以旧表为准覆盖
            if (!Objects.equals(mysqlEntity.getWithholdState(), obEntity.getWithholdState())
                    || !Objects.equals(mysqlEntity.getWithholdAmount(), obEntity.getWithholdAmount())
                    || !Objects.equals(mysqlEntity.getResultCode(), obEntity.getResultCode())
                    || !Objects.equals(mysqlEntity.getResultMsg(), obEntity.getResultMsg())
                    || !DateUtils.truncatedEquals(mysqlEntity.getDateUpdated(),
                            obEntity.getDateUpdated(), Calendar.SECOND)) {
                orderRepayWithholdObDao.updateByRepayTranNo(mysqlEntity);
                diffCount++;
            }
        }
    }

    startId = endId;
}

if (diffCount > 0) {
    log.info("PRO 二次校验 发现{}条关键字段不一致的记录，已以旧表数据覆盖写入OB", diffCount);
} else {
    log.info("PRO 二次校验 所有关键字段一致，数据一致");
}

6.4 抽样数据比对监控

迁移完成后，新增定时任务 OrderRepayWithholdDataCompareTask，按ID等间隔抽样比对 MySQL 旧表与 OceanBase 表的数据，发现差异时输出异常日志告警。

设计要点
  • 抽样策略：从起始ID开始，每隔一定间隔（如每1000条）抽取1条数据进行比对，兼顾覆盖率与执行效率
  • 比对维度：以 id 定位记录，对比关键字段（withhold_state、withhold_amount、result_code、result_msg、date_updated 等），不做全字段对比以减少内存占用
  • 比对表：旧MySQL表 vs OB表，验证迁移和三写的数据一致性
  • 触发条件：仅在迁移完成后（migration_status = COMPLETED）且写入模式为 TRIPLE 或 DUAL 时运行
  • 异常处理：发现差异时输出 WAN 级别日志（业务告警），不抛异常、不阻断主流程

任务实现

@Slf4j
@Service("orderRepayWithholdDataCompareTask")
public class OrderRepayWithholdDataCompareTask
        extends AbstractApsExecuteScheduledService {

    private static final int SAMPLE_INTERVAL = 1000;  // 每隔1000条抽1条

    @Resource
    private OrderRepayWithholdDao orderRepayWithholdDao;              // 旧MySQL
    @Resource
    private OrderRepayWithholdObDao orderRepayWithholdObDao;          // OB
    @Resource
    private OrderRepayWithholdMigrationRecordService migrationRecordService;

    @Override
    public List<ExecuteItem> selectTasks(TaskParam taskParam, String ownSign) {
        // 前置条件：迁移未完成 或 非三写/双写模式，不执行
        if (!migrationRecordService.isMigrationCompleted()
                || getWriteMode() == OLD_ONLY) {
            log.info("PRO 抽样数据比对 前置条件不满足，跳过执行");
            return null;
        }

        // 获取旧表ID范围
        Long minId = orderRepayWithholdDao.selectMinId();
        Long maxId = orderRepayWithholdDao.selectMaxId();
        if (minId == null || maxId == null) {
            log.info("PRO 抽样数据比对 旧表无数据");
            return null;
        }

        log.info("PRO 抽样数据比对 开始 minId={} maxId={} interval={}", minId, maxId, SAMPLE_INTERVAL);

        // 按间隔抽样比对
        int diffCount = 0;
        int sampleCount = 0;
        for (long id = minId; id <= maxId; id += SAMPLE_INTERVAL) {
            sampleCount++;

            // 查旧MySQL
            OrderRepayWithholdEntity mysqlEntity =
                orderRepayWithholdDao.queryById(id);
            if (mysqlEntity == null) {
                continue;  // 该ID不存在，跳过
            }

            // 查OB
            OrderRepayWithholdEntity obEntity =
                orderRepayWithholdObDao.selectByRepayTranNo(mysqlEntity.getRepayTranNo());

            if (obEntity == null) {
                log.warn("WAN 抽样数据比对 OB缺失 id={} repayTranNo={} withholdState={}",
                    id, mysqlEntity.getRepayTranNo(), mysqlEntity.getWithholdState());
                diffCount++;
                continue;
            }

            // 比对关键字段
            if (!Objects.equals(mysqlEntity.getWithholdState(), obEntity.getWithholdState())
                    || !Objects.equals(mysqlEntity.getWithholdAmount(), obEntity.getWithholdAmount())
                    || !Objects.equals(mysqlEntity.getResultCode(), obEntity.getResultCode())
                    || !Objects.equals(mysqlEntity.getResultMsg(), obEntity.getResultMsg())
                    || !DateUtils.truncatedEquals(mysqlEntity.getDateUpdated(),
                            obEntity.getDateUpdated(), Calendar.SECOND)) {
                log.warn("WAN 抽样数据比对 字段不一致 id={} repayTranNo={} " +
                    "mysql[state={},amount={},resultCode={},dateUpdated={}] " +
                    "ob[state={},amount={},resultCode={},dateUpdated={}]",
                    id, mysqlEntity.getRepayTranNo(),
                    mysqlEntity.getWithholdState(), mysqlEntity.getWithholdAmount(),
                    mysqlEntity.getResultCode(), mysqlEntity.getDateUpdated(),
                    obEntity.getWithholdState(), obEntity.getWithholdAmount(),
                    obEntity.getResultCode(), obEntity.getDateUpdated());
                diffCount++;
            }
        }

        log.info("PRO 抽样数据比对 完成 sampleCount={} diffCount={}", sampleCount, diffCount);
        return null;
    }
}

10. 已知问题与风险

10.1 旧表时间范围查询效率

MySQL 旧表 order_repay_withhold 当前无 date_updated 索引，按 date_updated 时间范围扫描旧表会导致全表扫描，效率极低。本改造已通过以下方式规避该问题：

规避方案

场景原方案（全表扫描）现方案（走主键索引）
抽样比对监控（6.4节） 按 date_updated 时间范围查全量 按 ID 等间隔抽样，通过 queryById 点查
二次校验（6.2节） 按 date_created <= ? AND date_updated >= ? 查全量 按 ID 范围分批扫描，逐条比对 date_updated 内存过滤
结论 二次校验和抽样比对均已改为按 ID 扫描方案，走主键索引，性能可控。当前改造不需要在旧表上新增 date_updated 索引。如果未来有其他场景需要按 date_updated 高效查询，再考虑使用 pt-online-schema-change 在线加索引。

10.2 分区表主键必须包含分区键

MySQL 分区表要求主键（及所有唯一键）必须包含分区键。当前设计中热表 order_repay_withhold_hot 按 date_created 做 RANGE COLUMNS 分区，因此主键必须包含 date_created，这带来了以下影响：

当前设计中的主键调整

项目旧表（非分区）新热表（分区）
主键定义 PRIMARY KEY (id) PRIMARY KEY (date_created, id)
id 列属性 AUTO_INCREMENT + PRIMARY KEY AUTO_INCREMENT（非主键成员在前），通过 idx_id 二级索引保证按 id 查询性能
唯一键 UNIQUE KEY (repay_tran_no) UNIQUE KEY (repay_tran_no, date_created)

带来的影响

  1. id 不再是独立主键：id 保留 AUTO_INCREMENT 但不再是 PRIMARY KEY 的前缀列，按 id 的点查需走二级索引 idx_id 回表，比直接走主键多一次回表开销
  2. 唯一键需追加分区键：uk_repay_tran_no 必须变为 (repay_tran_no, date_created)，这意味着同一条 repay_tran_no 在不同日期可以存在多条记录（理论上不应出现，但约束变弱）
  3. 所有二级索引均需包含 date_created：MySQL 5.6 分区表的二级索引自动包含分区键，索引体积增大，写入性能略有下降
  4. TRIPLE 模式下 id 一致性：旧表 id 为自增主键，热表 id 为非主键自增列。热表 insertSelective 在 TRIPLE 模式下使用旧表回填的 id 值插入，需确保热表 AUTO_INCREMENT 起始值大于旧表当前最大 id，避免自增序列冲突
应对措施
  • 热表建表后，执行 ALTER TABLE order_repay_withhold_hot AUTO_INCREMENT = {旧表当前最大id + 1}，确保 TRIPLE 模式下热表自增序列不会与旧表回填的 id 冲突
  • 按 id 查询的场景（如迁移任务基于 ID 范围扫描）走 idx_id 二级索引，性能可接受
  • 业务上 repay_tran_no 全局唯一，(repay_tran_no, date_created) 联合唯一键在实际业务中不会出现同号不同日的数据，约束变弱仅是理论风险
