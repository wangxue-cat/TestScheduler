# analyze-failure 详细规则

## 1. 证据要求

根因分析必须有证据支撑（日志/响应/数据），confidence 低于 "low" 的标记为 unknown。

## 2. 不提 Bug 的类别

`test_data` 和 `env_issue` 类别的失败不提 Bug（属于测试环境问题）。

## 3. 提 Bug 的类别

`api_logic` 和 `param_assembly` 类别的失败生成 Bug 草稿。

## 4. Bug 去重

每个失败最多生成一个 Bug 草稿（同一 case+step 的去重）。

## 5. 根因分类表

| 类别 | 说明 | 示例 |
|------|------|------|
| `test_data` | 测试数据问题 | account_id 过期/不存在 |
| `param_assembly` | 参数组装错误 | amount 未正确传递 |
| `api_logic` | 接口业务逻辑 Bug | 授信拒绝条件判断错误 |
| `env_issue` | 环境问题 | 服务不可达/配置不一致 |
| `downstream` | 下游服务异常 | 360 API 超时 |
| `unknown` | 无法定位 | 需要更多信息 |

## 6. 逐步骤分析维度

**a) 响应码分析**
- code ≠ 000000：检查是否为已知错误码
- code = 999999：通常是系统异常，需查日志
- 超时/网络错误：判断环境还是服务问题

**b) 字段值比对**
- expected vs actual 差异
- 判断参数/业务逻辑/测试数据问题

**c) 日志关联分析**
- 错误日志与响应关联
- 提取错误堆栈中的关键类/方法
- 判断错误发生在哪一层（参数校验/业务逻辑/下游调用/数据库）

**d) io_bindings 链路追踪**
- 检查上游步骤是否正常
- 判断是否数据传递问题
