# analyze-code-for-testcase

分析代码变更内容，结合需求材料，生成结构化的测试点清单（Happy Path / 边界 / 数据流 / 异常处理），辅助 Test Case Writer 编写更全面的测试用例。

## 输入

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| requirement_id | string | 是 | 需求编号 |
| code_diff_json | object | 否 | 已获取的 code diff，不提供则自动调 fetch-code-diff |
| requirement_materials_path | string | 否 | 需求材料目录路径 |

## 执行步骤

### Step 1: 确保输入就绪

1. 若未提供 `code_diff_json`，调用 `fetch-code-diff` 获取
2. 加载需求材料：
   - `{req_id}_requirement.md` — 需求描述和验收标准
   - `{req_id}_dev_doc.md` — 开发设计文档（若存在）
   - `{req_id}_tester_notes.md` — 测试说明（若存在）
3. 提取验收标准条目作为测试目标

### Step 2: 代码变更深度分析

对每个变更文件，调 `Skill(testmind-facade)` → `analyze-code-change`，获取：
- 变更范围（哪些方法/类被修改）
- 变更类型（新增功能 / Bug修复 / 重构 / 配置变更）
- 上下游依赖影响
- 关键逻辑路径

### Step 3: 生成 Happy Path 测试点

针对每个需求验收标准条目，从代码中还原其实现路径：

```
入口方法 → 参数校验 → 业务逻辑 → 外部调用 → 数据持久化 → 返回响应
```

每个路径节点生成一条 Happy Path 测试点：
- **前置条件**：触发该路径需要的状态和数据
- **输入参数**：正常值范围
- **预期结果**：返回值、DB 落库、日志输出
- **代码位置**：具体文件:方法:大致行号

### Step 4: 生成边界场景测试点

按以下规则扫描代码，对每个潜在边界生成测试点：

| 边界类型 | 检测规则 | 示例 |
|---------|---------|------|
| null 参数 | 方法参数为引用类型，代码中未做 null check | `method(String name)` 无 `if (name == null)` |
| 空集合 | 参数为 List/Set/Map，代码中直接遍历 | `for (Item i : list)` 前无 `isEmpty()` 检查 |
| 数值边界 | 参数为数值类型，用于比较或计算 | `amount > 0` → 测试 amount=0, -1, MAX_VALUE |
| 字符串长度 | 参数为 String，用于截取或比较 | `substring(0, 10)` → 测试 0 长度、超长 |
| 并发场景 | 代码中有锁/synchronized/并发集合 | 测试并发写入同一记录 |
| 超时场景 | 代码中有外部 HTTP/RPC 调用 | 测试外部服务超时/不可用 |

### Step 5: 生成数据流测试点

追踪关键数据从入口到持久化的完整路径：

1. **入参字段** → 识别 API 入口的参数对象
2. **校验层** → 参数校验/转换逻辑
3. **业务层** → 业务逻辑中的数据变换
4. **持久化层** → 写入的 DB 表和字段
5. **响应层** → 返回给调用方的数据

每个数据流路径生成测试点，标注每个阶段的字段名和变换逻辑。

### Step 6: 生成异常处理测试点

扫描代码中的异常路径：

- `throw` 语句 → 测试异常触发条件
- `catch` 块 → 测试异常处理逻辑
- 错误码返回 → 测试错误场景
- `@Transactional` rollback → 测试事务回滚
- finally 块 → 测试资源释放

### Step 7: 补充代码级建议

调 `Skill(testmind-facade)` → `testcase-code-analyze` 获取额外的代码覆盖建议。

### Step 8: 生成报告

## 输出

输出文件：`memory/code_analysis/{req_id}/testcase_aid_report.md`

报告结构：

```
# {req_id} 代码分析 — 测试用例辅助

> 需求: {title} | 分析时间: {date}
> 仓库: {repo} | 分支: {branch}
> 变更文件: N 个 | 变更方法: M 个

## 1. 变更概览

| 文件 | 变更类型 | 行数(+/-) | 关键变更方法 | 关联需求条目 |
|------|---------|-----------|-------------|-------------|

## 2. Happy Path 测试点

### 需求条目 X: {描述}

| TP-ID | 测试点描述 | 代码路径 | 前置条件 | 预期结果 | 验证方法 |
|-------|-----------|---------|---------|---------|---------|
| HP-001 | ... | File.method():line | ... | ... | 接口调用/DB查询/日志 |

## 3. 边界场景测试点

| TP-ID | 边界类型 | 测试点描述 | 代码位置 | 边界条件 | 预期行为 |
|-------|---------|-----------|---------|---------|---------|
| BC-001 | null参数 | ... | File.method():line | param=null | 返回错误码/抛异常 |

## 4. 数据流测试点

| TP-ID | 数据路径 | 关键字段 | 变换节点 | 验证点 |
|-------|---------|---------|---------|--------|
| DF-001 | 入参→DB | fieldA, fieldB | Validator→Service→DAO | DB表.table.col |

## 5. 异常处理测试点

| TP-ID | 异常类型 | 触发条件 | 代码位置 | 预期处理 |
|-------|---------|---------|---------|---------|
| EH-001 | TimeoutException | RPC超时 | File.method():line | 降级返回/重试 |

## 6. 建议优先级

| 优先级 | 测试点数量 | 说明 |
|--------|----------|------|
| 高 | N | 核心路径 + 高风险边界 |
| 中 | N | 一般边界 + 数据流 |
| 低 | N | 罕见异常 + 极端边界 |

## 7. 给 Test Case Writer 的建议

- 需关注的复杂逻辑: ...
- 建议增加的验证维度: ...
- 代码中发现的隐含假设: ...
```

## 关键规则

1. **只分析 diff 范围内文件**：不扩展到未变更的代码
2. **每个测试点有代码位置引用**：格式 `文件:方法:大致行号`
3. **不写完整用例**：只写测试点和关键参数，不写成完整的 Excel 用例行
4. **置信度标注**：代码路径不确定时标注"推断"，供人工判断
5. **关注 API 测试视角**：从接口调用者的角度思考，而非单元测试视角
6. **区分通用和特有**：通用边界（如 null check）标注为通用，特有业务逻辑重点描述
7. **关联需求条目**：每个测试点标注关联的需求条目编号
