# analyze-code-for-testcase 详细规则

## 范围限制规则

1. **变更文件白名单**：只分析 `fetch-code-diff` 返回的 `changed_files` 列表中的文件
2. **排除文件类型**：
   - `*Test.java` / `*Tests.java` — 单元测试文件
   - `pom.xml` 仅版本号变更 — 无测试意义
   - `.gitignore` / `README.md` — 非代码文件
3. **包含文件类型**：
   - `*.java` — 业务代码
   - `*.xml` — Mapper/配置（含 SQL 变更）
   - `*.sql` — 数据库变更脚本
   - `*.vue` / `*.tsx` / `*.jsx` — 前端代码
   - `*.yml` / `*.properties` — 配置文件
4. **跨文件追踪**：若变更方法调用了未变更的工具方法，**可以**简短查看被调用方法签名，但**不深度分析**未变更代码

## 需求-代码映射方法

1. 从 `{req_id}_requirement.md` 提取验收标准条目
2. 对每个条目：
   - 提取关键词（业务实体名、操作名、状态名）
   - 在 diff 中搜索匹配的类名/方法名/注释
   - 找到匹配 → 标记为关联
   - 未找到 → 标记为"无对应代码变更"
3. 反向检查：diff 中是否有变更找不到对应的需求条目 → 标记为"未归类变更"

## 边界场景生成规则

### null 参数检测

对每个新增/修改的方法参数：
- 参数类型为引用类型（非基本类型）且方法体内无 `if (param == null)` → **高风险**：生成 null 测试点
- 参数有 `@NonNull` 注解 → **低风险**：标注但可不生成
- 参数在方法内直接调用方法（如 `param.foo()`）→ **极高风险**：必须生成

### 空集合检测

对每个遍历操作（`for`, `forEach`, `stream()`）：
- 遍历前无 `isEmpty()` 检查 → 生成空集合测试点
- 遍历的是外部传入的集合 → **高风险**
- 遍历的是方法内部 new 的集合 → **低风险**

### 数值边界检测

对每个数值比较操作：
- `> 0` / `>= 0` → 测试 0、-1
- `< MAX` → 测试 MAX、MAX+1（溢出）
- 金额计算 → 测试 0.00、0.01、超大金额
- 分页参数 → 测试 page=0、超大 pageNo

### 外部调用检测

对每个 HTTP/RPC/DB 调用：
- 无 try-catch 包裹 → 生成超时/异常测试点
- 有 try-catch → 查看 catch 块逻辑，生成异常处理测试点
- 有降级逻辑 → 生成降级触发测试点

## 数据流追踪规则

### 追踪路径

```
Controller/Facade 入口
  → 参数校验（Validator/Checker）
    → DTO 转换（Converter/Assembler）
      → 业务逻辑（Service/Manager）
        → 外部调用（RPC/HTTP/MQ）
        → 数据持久化（DAO/Repository/Mapper）
      → 响应组装（ResponseBuilder）
    → 返回
```

### 字段关注点

1. **身份标识字段**：userId, orderId, reqNo → 全程追踪
2. **金额字段**：amount, fee, interest → 关注计算和单位转换
3. **状态字段**：status, state, type → 关注枚举值和状态机
4. **时间字段**：createTime, updateTime → 关注时区和格式

### 变换节点

- 枚举值映射（如：接口传 `"SUCCESS"` → DB 存 `1`）
- 金额单位转换（元 → 分）
- 字段名映射（如：接口 `repay_order_id` → DB `repay_tran_no`）
- JSON 序列化/反序列化

## 置信度评分标准

| 置信度 | 条件 | 表现 |
|--------|------|------|
| **高** | 代码路径清晰，可直接追踪到完整调用链 | 可直接采用 |
| **中** | 有合理的业务推断，但缺少部分环节的确认 | 标注"推断"，建议人工确认 |
| **低** | 依赖命名约定或通用的代码模式推断 | 明确标注"推测"，必须人工确认 |

## 与 Test Case Writer 的协作接口

生成的报告应包含足够的信息供 testcase-writer 直接使用：

1. **前置条件**：可直接写入测试用例的「前置条件」列
2. **输入参数**：可直接写入测试用例的「输入参数」列
3. **预期结果**：可直接写入测试用例的「预期结果」列
4. **验证方法**：标注是 API 返回校验、DB 落库校验、还是日志关键字校验
