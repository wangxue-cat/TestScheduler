# verify-bug-fix

从 Bug 描述和修复代码出发，验证修复是否真正解决了根因而非掩盖症状，检查修复引入的副作用和回归风险，建议补充测试点。

## 输入

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| bug_id | string | 是 | Bug 编号（如 JYSG-150160） |
| code_diff_json | object | 否 | 已获取的 code diff |
| bug_info | object | 否 | Bug 信息（来自 result-analyst 的 analyze-failure 输出或 bug-manage 查询结果） |

## 执行步骤

### Step 1: 加载 Bug 信息

1. 若提供了 `bug_info`，直接使用
2. 否则调用 `Skill(testmind-facade)` → `bug-manage` 查询 Bug 详情：
   - Bug 标题和描述
   - 严重程度
   - 复现步骤
   - 关联的需求/Story

3. 同时加载 result-analyst 的失败分析报告（若存在）：
   - `memory/test_results/reports/{req_id}_execution_report.md`
   - 或 `memory/test_results/reports/{req_id}_verification_report.md`

### Step 2: 获取修复 diff

1. 若提供了 `code_diff_json`，直接使用
2. 否则调用 `fetch-code-diff`，传入 `bug_id`

### Step 3: 根因分析评估

对照 Bug 描述和代码 diff，回答以下问题：

| 问题 | 评估 |
|------|------|
| 修复是否针对根因？ | 根因 / 症状 / 无法判断 |
| 修复位置与失败分析一致？ | 一致 / 不同位置 / 无失败分析 |
| 修复方案是否最小化？ | 最小 / 附带重构 / 过度修改 |
| 是否有更简单的修复方案？ | 无 / 有（描述） |

### Step 4: 修复质量评估

| 维度 | 评估要点 |
|------|---------|
| **正确性** | 修复逻辑是否正确覆盖了所有触发场景 |
| **完备性** | 是否只修复了一处，但同样的问题在其他地方也存在 |
| **防御性** | 是否添加了足够的边界检查和异常处理 |
| **可读性** | 修复代码是否清晰，还是引入了晦涩的逻辑 |

### Step 5: 副作用分析

调用 `Skill(testmind-facade)` → `code-info` 查询：
1. 变更方法的所有调用方
2. 变更方法的签名是否变化（影响调用方）
3. 变更逻辑是否影响共享的其他代码路径

分析维度：
- **调用方影响**：接口变更/行为变更/返回值变化
- **共享逻辑影响**：被多个路径调用的方法是否因修复产生预期外的影响
- **数据流影响**：DB 字段/缓存/MQ 消息是否受影响
- **新增风险**：修复是否引入了新的异常/边界/并发问题

### Step 6: 回归风险评估

综合以下因素评估回归风险等级：

| 因素 | 高风险信号 |
|------|-----------|
| 变更范围 | 修改方法超过 5 个，或核心公共方法 |
| 调用方数量 | 变更方法有 10+ 调用方 |
| 方法可见度 | 修改了 public/interface 方法签名 |
| 变更类型 | 修改了核心业务逻辑而非新增检查 |
| DB 变更 | 涉及表结构或 SQL 逻辑变更 |

**风险等级**：High / Medium / Low

### Step 7: 生成补充测试点

| 测试点类型 | 说明 |
|-----------|------|
| **原始复现** | 复现原始 Bug 场景，确认修复后不再出现 |
| **修复验证** | 验证修复逻辑在正常和边界条件下的正确性 |
| **回归测试** | 验证修复影响的调用方的原有功能不受影响 |
| **同类问题** | 代码库中是否存在同样的模式缺陷 |

### Step 8: 输出验证结论

## 关键规则

1. **结论分级明确**：
   - `verified_correct` — 修复正确解决根因，无副作用，无回归风险
   - `verified_partial` — 修复部分解决根因，或存在轻微副作用
   - `verified_incorrect` — 修复未解决根因，或引入严重副作用
   - `cannot_verify` — 信息不足，无法给出结论（需说明缺什么信息）
2. **不重复 analyst 工作**：若 result-analyst 已有失败分析，引用其结论而非重新分析
3. **副作用分析必做**：不管修复多小，都要检查调用方和共享逻辑
4. **补充测试点可执行**：不是抽象建议，是具体可写成测试步骤的点
5. **不能验证时诚实说明**：缺少信息 → `cannot_verify` + 列出需要的信息
