# verify-bug-fix 详细规则

## 根因验证准则

### 判断是否为根因修复

| 修复类型 | 判断 | 说明 |
|---------|------|------|
| 修复了导致错误的逻辑判断 | ✅ 根因 | 如：`if (a > b)` 改为 `if (a >= b)` |
| 添加了之前缺失的 null check | ✅ 根因 | 如：增加了 `if (obj == null) return;` |
| 修复了数据转换逻辑 | ✅ 根因 | 如：金额单位从分改为元 |
| 捕获了之前未处理的异常 | ⚠️ 可能是症状 | 若底层异常原因未解决，只是不报错了 |
| 增加了重试逻辑 | ⚠️ 可能是症状 | 若下游服务不稳定问题未解决，重试是掩盖 |
| 只修改了日志/错误消息 | ❌ 症状 | 不是修复，只是美化 |
| 回滚了引入问题的 commit | ✅ 根因 | 但需确认不会丢失其他功能 |

### 修复位置一致性

- 修复代码的位置与 result-analyst 定位的失败点一致 → 高信心
- 修复代码在完全不同的模块 → 需要验证是否真的相关
- result-analyst 没有做失败分析 → 无法对比，需要从 Bug 描述推断

## 修复质量评估标准

### 正确性

- 修复逻辑是否覆盖了 Bug 描述中所有的复现场景？
- 是否有遗漏的触发条件？
- 边界条件是否也一并处理了？

### 完备性

- 同样的问题是否在代码库其他地方也存在？
- 修复者是否只修了报出来的那个场景，忽略了同模式的其他地方？
- 提示方式：搜索相似的代码模式 → 若其他地方也有同样的问题 → `verified_partial`

### 防御性

- 修复是否只在单一检查点防护？
- 是否应该在上游做更早的校验？
- 是否应该在多个层级（Controller + Service + DAO）都有防护？

### 可读性

- 修复代码是否有注释说明为什么这样修？
- 逻辑是否清晰，还是引入了晦涩的 workaround？

## 副作用检测矩阵

### 方法签名变更影响

| 变更类型 | 影响范围 | 检测方法 |
|---------|---------|---------|
| 方法参数增加 | 所有调用方需传新参数 | 查 code-info 找所有调用方 |
| 方法参数类型变更 | 所有调用方编译失败 | 查 code-info 找所有调用方 |
| 返回值类型变更 | 所有调用方需要调整 | 查 code-info 找所有调用方 |
| 异常声明变更 | 调用方 catch 块可能失效 | 查调用方的 catch 逻辑 |
| 方法可见度变更 | 外部模块调用失效 | 查跨模块引用 |

### 行为语义变更影响

| 变更类型 | 影响范围 | 检测方法 |
|---------|---------|---------|
| 返回值含义变化 | 依赖旧语义的调用方出问题 | 审查调用方如何使用返回值 |
| 异常抛出时机变化 | 依赖旧异常行为的调用方 | 审查调用方的 catch 逻辑 |
| 副作用时机变化 | 依赖特定执行顺序的逻辑 | 审查并发/时序相关代码 |
| null 返回变为抛异常 | 依赖 null 判断的逻辑 | 审查调用方的 null check |

### DB 变更影响

| 变更类型 | 影响范围 | 检测方法 |
|---------|---------|---------|
| SQL 修改 | 影响查询结果/写入数据 | 审查 Mapper XML / JPA 注解 |
| 表字段新增 | 通常无影响，但需确认默认值 | 审查实体类和 DDL |
| 表字段删除 | 查询/写入失败 | 审查所有引用该字段的代码 |
| 索引变更 | 查询性能变化 | 审查查询条件 |

## 回归风险评估

### 评分表

| 因素 | 权重 | 低风险(0) | 中风险(1) | 高风险(2) |
|------|------|----------|----------|----------|
| 变更文件数 | 20% | 1-2 | 3-5 | 6+ |
| 变更方法可见度 | 25% | private | protected/package | public |
| 调用方数量 | 25% | 0-3 | 4-10 | 10+ |
| DB/SQL 变更 | 15% | 无 | 查询变更 | 写入/DDL 变更 |
| 核心业务路径 | 15% | 非核心 | 次要路径 | 核心支付/授信路径 |

**总分**: < 0.5 → Low / 0.5-1.0 → Medium / > 1.0 → High

### 高风险信号（单个因素即可判定 High）

- 修改了金额计算逻辑
- 修改了事务边界
- 修改了分布式锁逻辑
- 修改了支付/授信状态机
- 修改了公共基础库方法

## Bug 修复测试点生成规则

### 原始复现（必生成，≥1 条）

- 从 Bug 描述提取复现步骤
- 转换为可执行的测试步骤
- 期望：修复后同样的步骤应该得到正确的结果

### 修复验证（必生成，≥2 条）

- 正常场景验证修复逻辑
- 边界场景验证修复逻辑
- 异常场景验证修复逻辑

### 回归测试（根据影响范围生成）

- 对每个受影响的调用方生成 1 条回归点
- 对每个共享逻辑路径生成 1 条回归点
- 对 DB 变更生成数据一致性检查点

### 同类问题扫描（条件生成）

- 若修复的是一个模式缺陷（如某类空检查缺失），在代码库中搜索相似模式
- 找到相似模式 → 生成同类问题验证点
- 未找到 → 标注"未发现同类问题"
