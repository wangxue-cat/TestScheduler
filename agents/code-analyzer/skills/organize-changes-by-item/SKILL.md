# organize-changes-by-item

扫描代码 diff + 需求文档，将变更文件/方法映射到需求条目，按条目组织输出结构化的代码改动清单。

## 输入

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| requirement_id | string | 是 | 需求编号（如 NREQUEST-48777） |
| code_diff_json | object | 否 | 已获取的 code diff，不提供则自动调用 fetch-code-diff |
| requirement_materials_path | string | 否 | 需求材料目录，默认 `memory/requirement_materials/{req_id}/` |

## 执行步骤

### Step 1: 确保输入就绪

1. 若未提供 `code_diff_json`，调用 `fetch-code-diff` 获取
2. 加载需求文档：`{req_id}_requirement.md`
3. 提取需求条目：
   - `## 验收标准` → 编号列表项
   - `## 功能点` / `## 需求描述` → 功能条目
   - 带编号的段落（1. 2. 3. ...）

### Step 2: 分析每个变更文件

对 `changed_files` 中的每个文件：

1. 调 `Skill(testmind-facade)` → `analyze-code-change` 获取变更分析
2. 获取变更的方法/类列表
3. 记录变更类型（新增/修改/删除）和行数

### Step 3: 映射变更到需求条目

按以下优先级将每个变更文件/方法映射到需求条目：

| 优先级 | 映射线索 | 置信度 |
|--------|---------|--------|
| 1 | Commit message 中引用了需求条目编号 | 高 |
| 2 | 方法名/类名匹配需求中的业务术语 | 高 |
| 3 | 代码注释中引用了需求条目 | 中 |
| 4 | 变更范围与需求条目功能描述相关 | 中 |
| 5 | 同模块/同 package 的其他变更已确定映射 | 低 |

### Step 4: 处理未归类变更

标记无法映射到任何需求条目的变更：
- 单独列入「未归类变更」板块
- 标注"需人工确认是否属于本需求范围"
- 猜测最可能关联的需求条目（标注为推断）

### Step 5: 生成改动清单报告

## 输出

输出文件：`memory/code_analysis/{req_id}/change_by_item.md`

报告结构：

```
# {req_id} 代码改动清单

> 需求: {title} | 分析时间: {date} | 仓库: {repo} | 分支: {branch}

## 概览

| 指标 | 数值 |
|------|------|
| 需求条目数 | N |
| 变更文件数 | N |
| 变更方法数 | N |
| 已归类变更 | N |
| 未归类变更 | N |

## 按需求条目整理的改动

### 条目 1: {需求条目描述}

**代码改动**：

| 文件 | 方法/类 | 变更类型 | 行数(+/-) | 说明 |
|------|---------|---------|-----------|------|
| path/File.java | methodName() | 修改 | +15/-3 | 改动说明 |

**改动逻辑**：{简述此条目下所有改动的业务逻辑}

### 条目 2: ...

## 未归类变更

> ⚠️ 以下变更无法自动映射到需求条目，需人工确认

| 文件 | 方法/类 | 变更类型 | 可能关联条目 | 依据 |
|------|---------|---------|-------------|------|

## 未变更的需求条目

> ⚠️ 以下需求条目在代码中没有找到对应的变更

| 需求条目 | 可能原因 |
|---------|---------|
| 条目 N | 纯配置变更 / 前端变更 / 本次未实现 |

## 变更文件完整列表

| 文件路径 | 变更类型 | 行数(+/-) | 关联条目 |
|----------|---------|-----------|---------|
```

## 关键规则

1. **未归类变更必须标记**：不能默默忽略，必须放在显眼的区块
2. **未变更条目必须列出**：需求条目无对应代码变更时，标注可能原因
3. **置信度透明**：低置信度映射必须标注为「推断」，不伪装成确定结论
4. **复用现有分析**：若已在同一会话中执行过 fetch-code-diff，直接使用已有结果
5. **条目粒度**：需求条目拆分到合适的粒度（通常 3-8 个条目），不要太粗也不要太细
6. **代码位置引用**：每行必须有具体文件路径，方法级标注时加上大致行号范围
