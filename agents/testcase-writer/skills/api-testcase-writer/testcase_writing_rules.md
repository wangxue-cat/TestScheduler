# API测试用例编写规则

> 本文件定义用例编写的强制规则。用户可随时修改，skill执行时将读取最新版本。

## Rule 1: 执行步骤动作必须明确

步骤描述的核心要求是**动作明确、可执行**，不限制具体措辞形式。常见模式：

| 步骤模式 | 数据来源 | 示例 |
|---------|---------|------|
| 调用xx接口 | 接口文档 | 调用bindCard接口 |
| 执行xx流程（可含多步） | 接口文档 / 需求文档 | 提交借款流程（含试算→确认→签约） |
| 执行xx定时任务 | 开发设计文档 / 测试人员补充 / 代码 | 执行授信结果回调定时任务 |
| 数据库配置xx修改为xx | 开发设计文档 / 测试人员补充 / 代码 | 数据库配置max_retry修改为3 |
| apollo配置xx修改为xx | 开发设计文档 / 测试人员补充 / 代码 | apollo配置credit.switch修改为true |

**要点**：
- 单接口调用写"调用xx接口"，多步骤串联可写"执行xx流程"
- 动作必须让执行者明确知道要做什么，不可模糊如"测试授信"
- 默认行为：若设计文档、测试人员补充、代码中均无配置/任务信息，步骤仅包含接口调用或流程

**接口名确定方式**（不再从 `api_channels/*.json` 获取，直接从接口文档匹配）：
1. 调用 `run.py find-interface-doc` 根据渠道名在 `D:/TestScheduler/memory/interface_docs/` 中查找已缓存的 MD 文件
2. 若 MD 缓存未命中，自动在 `D:/接口文档合集/` 中查找原始接口文档（xlsx/docx/pdf），首次识别时自动转换为 MD 并缓存到 `memory/interface_docs/`
3. 若接口文档和 MD 缓存均未找到 → **必须提示用户提供接口文档**，不可跳过
4. 仅当确认是公共改造无接口文档时，步骤中描述具体操作即可

## Rule 2: 预期结果必须可验证

预期结果只能是以下形式之一：

| 预期结果模式 | 数据来源（严格限制） | 示例 |
|-------------|-------------------|------|
| xx接口的xx字段返回xx状态 | **仅限**：需求文档、接口文档、测试人员补充 | bindCard接口的code字段返回0状态 |
| xx接口返回响应成功 | **仅限**：需求文档、接口文档、测试人员补充 | pushOrderInfo接口返回响应成功 |

**关键限制**：接口返回字段断言**禁止**从开发设计文档或代码取值。原因：开发者可能实现有误，用设计文档的预期值做断言等于用开发者的假设验证开发者自身代码，无法发现bug。

> **落库预期规则**：若需求或用户明确提供了数据库落库、更新、删除等数据变更情况，**必须在预期结果中写出落库检查**，格式为 `xx库.xx表落库xx条记录，字段xx值为xx` 或 `xx库.xx表的状态变为xxx`。若需求未提及任何数据变更，则落库校验不在用例中编写，由 result-checker Agent 的 db-validation skill 单独处理。

## Rule 3: 渠道处理

| 场景 | 处理方式 |
|------|---------|
| 需求指定了渠道 | 在"前置条件"列添加渠道编码和名称，如 `渠道: PingAnPh(平安普惠)` |
| 需求涉及多个渠道 | 每个渠道生成独立用例集，每个用例标题和步骤必须不同（如"a渠道提交授信"、"b渠道提交授信"），因为上传规则拒绝重复用例名和步骤 |
| 需求未指定渠道 | 先查测试人员补充材料；若仍无 → **必须询问用户指定渠道**，不可自行假设 |

## 类型默认值规则

- **所有用例的"类型"字段默认填"功能测试"**，除非明确是纯接口级测试（类型填"接口"）或跨接口流程测试（类型填"流程"）
- 实际使用中，绝大多数用例都是功能测试，因此默认值简化为"功能测试"，减少编写时的判断成本

## 是否需要回归规则

- **"是否需要回归"列**：表示在所有测试完成后，拉了 release 版本时，是否需要再回归一次
- �域：`是` / `否`
- **通常只有本次需求的主流程用例标记为"是"**，其他校验类/边界类/异常类用例标记为"否"
- 判断标准：
  - 主流程（高优先级的核心正向流程）→ `是`
  - 重要校验逻辑（中优先级中与核心业务强相关的）→ 视情况而定，一般 `否`
  - 边界值/异常/兼容性（低优先级）→ `否`
- 示例：平安普惠授信正常流程用例 → `是`；平安普惠授信传1个联系人校验 → `否`

## 用例命名规范

- 格式：`{渠道名}{业务场景}{测试点}`
- 示例：`平安普惠授信传1个联系人`、`平安普惠授信传2个联系人`、`平安普惠授信不传联系人`
- 多渠道时渠道名不可省略
- 用例名称不可重复（上传校验规则）

## 优先级分配

| 优先级 | 适用场景 |
|--------|---------|
| 高 | 核心业务主流程（正常提交成功） |
| 中 | 重要业务场景（关键校验、重要异常） |
| 低 | 边界值、次要异常、兼容性 |

## 材料访问规则

需求材料由 Requirement Collector 统一拉取，内容已存入 `memory/requirement_materials/{req_id}/` 的本地 MD 文件中。用例编写时按以下方式获取信息：

| 信息需求 | 获取方式 |
|---------|---------|
| 需求描述正文 | 读取 `{req_id}_requirement.md`（由 `testmind:get-request-content` 拉取） |
| 需求基本信息（标题/状态/迭代/Story） | 读取 `{req_id}_requirement.md` 或调用 `testmind:request-manage` |
| 开发设计文档 | 读取 `{req_id}_dev_doc.md`（由 `testmind:confluence` 拉取） |
| 测试人员补充 | 读取 `{req_id}_tester_notes.md` |
| 接口定义 | 调用 `run.py find-interface-doc` 从接口文档匹配 |
| 其他外部 URL | WebFetch |
| 本地路径 | Read |
